## 4. Decision-Making Process

### 4.1 Decision Authority

| Decision Type | Authority | Process |
|---------------|-----------|----------|
| Bug fixes | Developer + Lead | PR review |
| Features | Lead + Team | Discussion + vote |
| Architecture | Lead + Architects | RFC process |
| Release | Owner | Final approval |
| Breaking changes | Owner + Lead | Community feedback |

### 4.2 RFC (Request for Comments)

- Major features go through RFC
- Discussion period: 1 week
- Implementation: Core team vote
- Community feedback: Valued

## 6. Onboarding Process

### 6.1 New Contributor

1. **Welcome:** Introduction & guidelines
2. **Setup:** Development environment
3. **First issue:** "Good first issue" label
4. **Mentoring:** Pair with core member
5. **Autonomy:** Increase responsibilities

### 6.2 Core Team Member

1. **Probation:** 3-6 months active contribution
2. **Review:** Evaluation by lead
3. **Approval:** Owner decision
4. **Permissions:** Write access granted
5. **Responsibilities:** Clearly defined

---

## 7. Conflict Resolution

### 7.1 Dispute Process

1. **Discussion:** Direct conversation
2. **Mediation:** Lead developer involved
3. **Arbitration:** Project owner decision
4. **Escalation:** If needed, community council (TBD)

### 7.2 Code of Conduct

- Respectful communication
- No harassment or discrimination
- Inclusive community
- Violations reported to owner
- Consequences: Warning → suspension → removal

---

## 8. Leadership Succession

### 8.1 Planned Handoff

- Owner can step down anytime
- Successor identified 3 months before
- Transition period: 1 month
- Approval: Community vote (if needed)

### 8.2 Emergency Succession

- Acting lead: Next senior developer
- Permanent successor: Community election
- Voting: Core team + active contributors
